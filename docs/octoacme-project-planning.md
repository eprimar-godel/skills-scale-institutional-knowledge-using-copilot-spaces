# OctoAcme — Project Planning

## Purpose
Turn an approved initiative into an actionable plan and backlog for delivery.

## Objectives
- Break work into shippable increments
- Identify dependencies and risks
- Align timelines, releases, and responsibilities

## Activities
1. Kickoff meeting with stakeholders and delivery team
2. Create prioritized backlog with acceptance criteria (BA drives requirements; PdM prioritizes)
3. Estimate scope (T-shirt sizing or story points)
4. Define Definition of Done (DoD)
5. Identify dependencies and integration points
6. Create release plan and milestone map
7. UX/UI Designer delivers design specs for planned features before dev starts
8. Security Lead defines security requirements and integrates scanning into the pipeline
9. Technical Writer plans documentation scope and milestones alongside the release plan

## Key Roles at This Phase
| Role | Responsibility during Planning |
|---|---|
| PM | Owns the project plan, timeline, and milestone map |
| PdM | Leads backlog prioritization and definition of acceptance criteria |
| Business Analyst | Translates requirements into user stories; facilitates backlog refinement |
| UX/UI Designer | Delivers wireframes/specs for planned features; flags UX risks |
| Security Lead | Defines security requirements per feature; validates CI/CD security tooling |
| Technical Writer | Scopes documentation work; adds doc tasks to the backlog |
| Developers | Estimate effort; identify technical risks and dependencies |
| QA/Testing | Drafts the test plan and defines the DoD quality criteria |
| Stakeholder Representative | Reviews and approves scope priorities on behalf of their stakeholder group |

> See [OctoAcme Roles & Personas](octoacme-roles-and-personas.md) for full role definitions.
> Use the [RACI & Handoffs guide](octoacme-raci-and-handoffs.md) to assign accountability across roles.

## Sprint / Iteration Planning
- Timebox planning to agreed sprint length
- Pull items that meet DoD and have clear acceptance criteria
- Ensure team capacity is respected

## Risk & Dependency Management
- Capture in Risk Register:
  - ID, Description, Impact, Probability, Owner, Mitigation
- Mark cross-team dependencies in the project board and escalate during weekly syncs

## Planning Checklist
- [ ] Project kickoff held (PM, PdM, BA, Developers, QA, Security Lead, UX/UI Designer)
- [ ] Business Analyst has documented and refined requirements into user stories
- [ ] Backlog prioritized and estimated
- [ ] UX/UI Designer has delivered wireframes/specs for in-scope features
- [ ] Security Lead has reviewed security requirements and CI/CD scanning configuration
- [ ] Technical Writer has scoped documentation tasks and added them to the backlog
- [ ] Release timeline and milestones agreed
- [ ] Definition of Done documented
- [ ] Initial test plan / QA approach drafted
